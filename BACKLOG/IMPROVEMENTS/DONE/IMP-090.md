# IMP-090: Preload hero image par page (pas seulement homepage)

**Type:** Improvement
**Statut:** A faire
**Priorite:** Moyenne
**Complexite:** S
**Tags:** vitrine, perf, lcp
**Date creation:** 2026-02-27

---

## Description

Actuellement, seule la homepage a un `<link rel="preload">` pour son image hero (`entretien-2-1920w.webp`). Les autres pages (services, contact, villes, blog, a-propos) chargent leur hero image sans preload, ce qui degrade le LCP (Largest Contentful Paint).

Chaque page devrait preload son image hero specifique pour optimiser le LCP.

## Criteres d'acceptation
- [ ] Chaque page avec hero image a un preload correspondant
- [ ] Preload utilise le bon format (WebP) et le bon `as="image"`
- [ ] Pas de preload inutile (seulement la variante la plus grande utilisee)
- [ ] LCP ameliore sur les pages services et villes

## Fichiers concernes
- `apps/vitrine/src/app/layout.tsx` (preload global actuel)
- `apps/vitrine/src/components/ui/HeroSection.tsx` (ou metadata par page)
- Chaque `page.tsx` qui utilise un HeroSection

## Analyse / Approche
Option A : Deplacer le preload du layout vers chaque page.tsx individuellement.
Option B : Modifier HeroSection pour injecter un preload dynamique via `<Head>` ou metadata.
Option C : Utiliser la prop `priority` du composant HeroClient pour forcer `fetchPriority="high"` + `loading="eager"` (deja fait, mais ajouter le preload link).
